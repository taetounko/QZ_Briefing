# 아키텍처

## 현재 구조

```text
QZ_Briefing/
├── config/            설정 파일 위치
├── data/              수집·저장 데이터 위치
├── logs/              실행 로그 위치
├── reports/           생성된 브리핑 위치
├── src/qz_briefing/   애플리케이션 소스 패키지
│   └── kiwoom/        연결 타입과 연결 관리자
└── tests/             테스트 패키지
```

## 설계 원칙

- 데이터 수집값, 계산값, 해석문을 서로 분리한다.
- 계좌 정보는 읽기 전용 경계를 유지한다.
- 주문 및 자동매매 경로는 두지 않는다.
- 브리핑에는 기준시각과 데이터 누락 상태를 함께 전달한다.
- 작은 기능 단위로 구현하고 각 기능 구현 후 테스트한다.

## 계층 구조

1. **키움 OCX 어댑터**: 실제 QAxWidget의 연결상태 읽기와 연결 요청을 최소 인터페이스로 감싼다. 현재 미구현 상태다.
2. **연결 관리자**: 연결상태 감시, 끊김 감지, 재접속 지연과 최대 횟수를 관리한다.
3. **향후 데이터 수집기**: 연결 관리자가 제공하는 안정된 연결 경계 위에서 시장 데이터를 수집한다.
4. **브리핑 분석 엔진**: 수집된 데이터를 계산하고 해석문을 생성한다.

연결 관리자는 키움 OCX 호출과 브리핑 분석 사이의 독립 계층이다. 실제 연결 객체와 시계를 주입받으므로 실제 OCX나 시간 대기 없이 테스트할 수 있고, 향후 Qt QTimer에서 `tick()`을 호출하는 방식으로 결합할 수 있다.

## 향후 구성 경계

세부 데이터 모델과 데이터 흐름은 `TASKS.md`의 데이터 모델 설계 단계에서 정의한다. 현재 단계에서는 실제 QAxWidget 어댑터, 데이터 수집기와 분석 엔진을 구현하지 않는다.
